FAQ

Everything that's worth knowing before you stare at a stuck publish job. Click any question to expand.

Who runs which step?

The app uses two identities simultaneously — there is no single "as the user" or "as the SP".

  • You (via OBO token) — every read on Unity Catalog (SHOW SHARES, SHOW RECIPIENTS, DESCRIBE, SHOW GRANTS, DESCRIBE TABLE, information_schema PK lookup) and the privilege mutations on the share itself: GRANT SELECT ON SHARE (publish) and REVOKE SELECT ON SHARE (delete /finalize). UC enforces your ownership / ALTER privilege when these run.
  • App service principal — listing / starting SQL warehouses, submitting and polling the BDC publish/delete Job, reading the app's own deployment path. The Job itself runs as the SP: it only calls the BDC SDK, which doesn't depend on user identity.

The audit trail stays coherent: system.access.audit shows GRANT/REVOKE attributed to you, the BDC SDK calls attributed to the SP. The Activity tab merges both views.

Why does the app sometimes ask me to "Grant access to SP"?

The BDC publish Job runs as the app's service principal. For it to read the schemas of your share's source tables (to build the CSN) and serve data through Delta Sharing, the SP needs USE CATALOG on the catalog, USE SCHEMA on each schema, and SELECT on each table.

The Review tile detects the catalogs your share pulls from and offers a one-click "Grant access to SP" button. The GRANTs run as you via OBO, so you must hold MANAGE (or ownership) on the catalog. Idempotent — safe to re-run.

If you don't hold MANAGE, ask the catalog owner to run the same grants.

I got "GRANT failed: you need ALTER on the share". Now what?

You don't own (or hold ALTER on) the share you're trying to publish. The app fails fast before submitting any Job, so there is no partial state to clean up.

Options: have the share owner publish it, or have them grant you ALTER on the share with GRANT ALTER ON SHARE <share> TO `your.email@…`. Direct grants take effect immediately — refresh the dropdown.

Publish fails with "Secret does not exist with scope: sap-bdc-connect-sdk". What's wrong?

The recipient you're publishing to was created with authentication_type = TOKEN instead of OIDC_FEDERATION. Confirm with DESCRIBE RECIPIENT <recipient-name>.

The BDC Connect SDK takes two different code paths depending on the recipient's auth type. OIDC_FEDERATION recipients carry their BDC tenant binding inline — the SDK handles auth automatically. TOKEN recipients don't carry that binding, so the SDK falls back to reading tenant config from a workspace secret scope literally named sap-bdc-connect-sdk (with keys api_url and tenant). If that scope doesn't exist in the workspace, you get the error above.

Recommended fix: drop the recipient and recreate it as OIDC_FEDERATION via the SAP BDC onboarding flow. No workspace bootstrap needed and it joins the existing flow cleanly.

If you must keep TOKEN auth (e.g. workspace where OIDC federation isn't available), a workspace admin needs to bootstrap the scope once. Get api_url + tenant values from the BDC team, then:

databricks secrets create-scope sap-bdc-connect-sdk
databricks secrets put-secret sap-bdc-connect-sdk api_url
databricks secrets put-secret sap-bdc-connect-sdk tenant
databricks secrets put-acl sap-bdc-connect-sdk <app-sp-client-id> READ

Mixing TOKEN and OIDC_FEDERATION recipients in the same workspace works mechanically but is not recommended — the two auth flows resolve tenant config from different places (per-recipient vs. workspace-wide), and BDC sees the resulting publishes attributed to two different publisher identities, which fragments audit and lineage.

Why does my share show as "already published" when I never published?

It shouldn't anymore. The previous heuristic flagged any share with a SELECT grant on a bdc-connect-* recipient as "published to BDC", which conflated two separate operations: the Databricks-side GRANT and the BDC-side publish_data_product.

Detection now uses the activity log: a (share, recipient) pair is "published" iff the latest activity event for the pair is publish_job_succeeded. If you publish via the app, this is always accurate. Manual / pre-activity-log publishes will appear unpublished — re-running the publish is safe (the BDC SDK uses create_or_update_share, idempotent on the share name).

The Job succeeded but the BDC catalog is empty. Bug?

No. The BDC catalog ingestion polls Databricks on its own cadence — typically every ~15 minutes. The Job's SUCCESS state and the notebook's Step 4 OK: data product published log line are the authoritative signals that the publish worked.

If the catalog is still empty after ~30 minutes, check the Activity tab and the linked Job run — but in nearly every case it's just the polling delay.

What does Delete do, exactly?

Two steps, in order:

  • The BDC publish Job calls BdcConnectClient.delete_share(share) — this removes the BDC catalog entry, ORD descriptor, CSN schema, and revokes BDC-side recipient access. Runs as the SP.
  • If the BDC delete succeeds, the route runs REVOKE SELECT ON SHARE as you via OBO. The order matters: if the BDC delete fails, your consumers don't lose SELECT mid-read.

The Databricks share itself is preserved (and so are its recipient grants other than the one you deleted). To remove the share entirely, run DROP SHARE in DBSQL — the app deliberately doesn't expose that.

Can I see who published / deleted what?

Yes — the Activity tab. Each publish writes 2 rows (publish_submitted, publish_job_succeeded); each delete writes 3 (delete_submitted, delete_job_succeeded, delete_finalized). Failures land as *_grant_failed, *_job_failed, or delete_finalize_failed with the exact error message.

Default view: your own events. Members of the group named in BDC_ACTIVITY_ADMIN_GROUP in app.yaml (default: admins) see a "Show all users" toggle that drops the filter. Group membership is resolved via SCIM /Me against your OBO token, so it must be a UC-recognized group your account belongs to.

Underlying table: chosen at first run via the in-app setup wizard (Activity tab → "Set up activity logging") and persisted to a config file owned by the app SP, or set explicitly via BDC_ACTIVITY_TABLE in app.yaml for scripted deploys. The configured table is shown in the footer of the Activity tab. App SP writes; Admins (account-level) and account users read. You can query it directly from any DBSQL editor.

Why are VIEWs and MATERIALIZED VIEWs blocked?

The current BDC Connect SDK doesn't support them — publish_data_product errors out partway through. The app blocks the publish in the Review tile so the failure surfaces immediately rather than 30 seconds into a Job run.

If a future SDK adds support, removing the block in analyzeObjects() is a one-line change.

Do I really need to provide primary keys?

Only for tables that don't already have one defined in Unity Catalog. BDC requires every published entity to have a primary key — tables with a UC-defined PK are picked up natively. The Review tile shows a column picker only for tables that need it; selected columns get annotated key: True, notNull: True in the CSN schema before publish.

What permissions does the app SP actually need?
  • CAN_USE on at least one SQL warehouse — granted via the workspace's SQL Warehouses → Permissions UI (the app intentionally doesn't bind to a specific warehouse; users pick one from the dropdown at runtime)
  • CAN_RUN on the publish + unpublish notebooks at the workspace paths in app.yaml
  • USE CATALOG / USE SCHEMA / SELECT on each catalog/schema/table the share pulls from — granted on demand from the Review tile's "Grant access to SP" button
  • If the activity log is enabled: USE CATALOG on the catalog, USE SCHEMA on the schema, and SELECT + MODIFY on the activity Delta table itself. UC enforces the full privilege chain — granting only on the table is not enough; the writer fails silently with PERMISSION_DENIED visible in databricks apps logs.

Not workspace admin. The SP also doesn't need any UC privileges on the share/recipient — those are checked against your identity when the route runs GRANT/REVOKE via OBO.

Is the activity log mandatory? Where does it get configured?

It is enabled by default via a self-service wizard. The first time the Activity tab is opened with no destination configured, the app shows a setup card with two paths:

  • Provision a new table — pick a catalog where you have MANAGE, leave the schema/table defaults (or override), click Provision and enable. The wizard runs DDL as you (OBO) and grants the app SP USE CATALOG, USE SCHEMA, SELECT, and MODIFY. Workspaces without a main catalog are explicitly supported.
  • Adopt an existing table — if a catalog admin already provisioned an activity table for the app, enter its 3-part name and click Verify and use. The wizard probes the table as the app SP (SELECT, INSERT, DELETE on a probe row) and saves the path if the probe passes. No UC privileges are required from you beyond what the SP already holds on that table.

The wizard never asks you to edit app.yaml or redeploy. Resolution order on every request: BDC_ACTIVITY_TABLE env var (scripted-deploy override) → workspace config file the wizard writes → null (Activity tab opens onto the wizard).

To turn the log off entirely, set BDC_ACTIVITY_TABLE to a non-empty placeholder you don't grant the SP on, or delete the wizard's config file at /Workspace/Users/<app-sp>/.bdc-app/config.json and ignore the wizard. With logging effectively off, the published-state detection in the share dropdown becomes silent (no false positives, no published markers either).

Where do I file bugs / request features?

Open an issue on the project's GitHub repository. Include the share + recipient names, the run id from the Activity tab, and the exact error message if the publish/delete failed.
